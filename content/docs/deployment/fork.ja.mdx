# Template Fork Workflow（プランA）

本リポジトリは「機能テンプレートリポジトリ（Template Repo）」として位置づけられ、機能の開発、問題の修正を行い、fork を通じて本番プロジェクトに同期します。

## 目標

- 機能コードの安定的な反復：`src/` に配置
- コンテンツの独立した差し替え：`content/` + `public/` に配置
- 多言語コンテンツの独立メンテナンス：`content/locales/` に配置

## ディレクトリ規約（現在実装済み）

- `content/site/i18n.yml`：言語設定の唯一の情報源
- `content/locales/*.json`：多言語テキストの唯一の情報源
- `content/site/nav.yml`：ナビゲーション構造設定
- `content/site/docs-nav.yml`：ドキュメントサイドバー構造
- `content/site/features.yml`：トップページ Features 構造
- `content/site/site.yml`：サイトブランドおよび公開表示情報
- `content/site/menu.yml`：Dashboard/Admin メニュー構造設定

検証スクリプトは自動的にランタイム成果物を `src/generated/content-config/*.json` に生成します（手動で編集しないでください）。

`src/config/*` は現在「アダプターレイヤー」として機能し、コアコンテンツデータを保存しなくなりました。

## 推奨ブランチ戦略

### テンプレートリポジトリ（本リポジトリ）

- `main`：テンプレート安定ブランチ（各本番プロジェクトが同期可能）
- `feature/*`：新機能開発
- `fix/*`：バグ修正

### 本番プロジェクト（fork 後）

- upstream をテンプレートリポジトリに向けたまま維持
- 定期的にテンプレート機能を同期し、自身のコンテンツを上書きしない

```bash
git remote add upstream <template-repo-url>
git fetch upstream
git checkout main
git merge upstream/main
```

## コンテンツ差し替え手順（本番プロジェクト）

1. `content/` 配下のビジネスコンテンツを差し替え（ドキュメント、製品、多言語）
2. `public/` 配下のブランドアセットを差し替え（ロゴ、画像、動画）
3. `src/` 機能レイヤーを維持し、テンプレート機能との分岐コンフリクトを回避

## ベストプラクティス：コンフリクト回避戦略

### 核心的な課題
テンプレートリポジトリ（Repo A）が機能コードを更新し、同時に `content/` 内のサンプルドキュメントや画像を修正した場合、本番プロジェクト（Repo B）で `git merge upstream/main` を実行すると以下の問題が発生する可能性があります：
1. **コンフリクト（衝突）**：Repo B も同じファイルを修正している場合。
2. **上書き（Overwrite）**：Repo B で修正していないファイルが Repo A の新しいサンプルで上書きされる場合。
### 解決策：Git Attributes コンテンツ保護（推奨）
merge driver を定義し、`content` と `public` ディレクトリをマージする際は**常に現在のブランチ（Repo B）を優先**するように Git に指示し、Upstream のコンテンツ変更を自動的に無視します。
**1. 本番プロジェクトのルートで Merge Driver を設定：**
これは一度限りの設定です（ローカルで有効）：
```bash
git config merge.ours.driver true
```
**2. 本番プロジェクトのルートに `.gitattributes` ファイルを作成：**
このファイルをリポジトリにコミットし、チームメンバー間で戦略を共有します：
```text
content/**/* merge=ours
public/**/* merge=ours
```
**3. 効果の確認**
設定後、`git merge upstream/main` を実行すると：
- `src/` 配下のコード変更：**通常通りマージ**（新機能を取得）。
- `content/` 配下のコンテンツ変更：**Upstream の変更を自動破棄**し、独自のビジネスデータを保持。
### 代替案：手動でのコンフリクト解決
`.gitattributes` を設定したくない場合、コンフリクト発生時に以下のコマンドを使用して自側のバージョンを素早く保持できます：
```bash
git merge upstream/main
# content/ でコンフリクトが発生した場合
git checkout --ours content/
git checkout --ours public/
git add .
git commit -m "Merge upstream features but keep local content"
```

## コンフリクト最小化のヒント（日常開発）

- **構造の分離**：本番プロジェクトで `src/config/*` 構造を直接変更しないでください。このディレクトリはテンプレートの更新に従うべきです。
- **テキストの抽出**：すべてのビジネステキストを `content/locales/` に移動し、コンポーネント内へのハードコードを避けてください。
- **設定の集中**：ナビゲーションとサイト情報の変更を `content/site/*` に集中させてください。


## 新規ページコンテンツ規約

新規ページ（例：about/contact/将来のポータルページ）は以下の規約に統一してください：

1. `content/site/*.yml` にページ設定を作成（最低限 `metadata.titleKey` と `metadata.descriptionKey` を含む）
2. ページ UI テキストは翻訳キーのみを参照し、テキストをハードコードしない
3. ページメタデータは統一ヘルパーを使用：`src/lib/page-metadata.ts`
4. 翻訳は `content/locales/*.json` にのみ記述
5. コミット前に `npm run content:validate` を実行し、スキーマと翻訳キーの検証を確認

## コミット前保護（組み込み済み）

- コンテンツスキーマ検証：`npm run content:validate`
- テンプレートチェック：`npm run check:template`

Git コミット前自動検証（pre-commit）を有効にするには：

```bash
npm run hooks:install
```

有効にすると、各コミット時に自動的に `content:validate` が実行され、以下を優先的に検出します：

- 不足している locale ファイル
- 不整合な locale キー
- `content/site/*` 設定構造エラー

## Cloudflare Pages デプロイ注意事項

- ビルドパイプラインは `node scripts/*.mjs` を使用してキースクリプトを直接実行し、ネストされた `npm run` 呼び出しによる不確実性を回避しています。
- 推奨 Pages Build コマンド：`npm run pages:build`
- データマニフェストの生成のみ：`npm run build:data`
